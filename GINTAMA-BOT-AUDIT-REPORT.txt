────────────────────────────────────────────────────────────────────────────────
BUG 1 — push and pull are completely missing from WindowsHandler and LinuxHandler
File: agent/handlers/windows.py  |  agent/handlers/linux.py
Severity: HIGH — commands silently fail
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  You type:   laptop push ./report.pdf /home/user/docs/
  The agent receives the command, looks for a method called "push"
  on the WindowsHandler or LinuxHandler class, finds nothing,
  and returns: "Action 'push' is not implemented on windows"

  The user sees a red ✗ in the terminal.

WHY THIS IS A PROBLEM:
  "push" and "pull" are listed in the README as core features.
  They're in the allowed_actions list in agent.toml.
  They're in the command grammar. But the two most-used OS handlers
  don't have them. Only phone.py (ADB module) has them.

  The Android handler (Termux mode) also does not have push/pull
  implemented — only the ADB phone module has it.

WHAT'S NEEDED:
  Both WindowsHandler and LinuxHandler need push() and pull() methods
  that use Python's shutil.copy2() to move files locally, and have a
  proper path validation step before writing anything.


────────────────────────────────────────────────────────────────────────────────
BUG 2 — reboot and shutdown always return "success" even when they fail
File: agent/handlers/windows.py  |  agent/handlers/linux.py
Severity: HIGH — false success reported to user
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  The reboot() and shutdown() methods call subprocess.run() but
  never check the return code. The function immediately returns
  {"status": "success"} regardless of whether the OS command
  actually worked.

  Example in windows.py:
    subprocess.run(["shutdown", "/r", "/t", "10"], shell=False)
    return {"status": "success", "message": "Rebooting in 10 seconds"}

  If the user doesn't have permission to reboot, or the command
  isn't found, the terminal shows green ✓ but nothing happens.
  No error is ever reported.

WHY THIS IS A PROBLEM:
  These are destructive operations. The user trusts the ✓ feedback.
  If you're doing a remote reboot and it silently fails, you have no
  idea the machine is still running.

WHAT'S NEEDED:
  Capture return code. If non-zero, return error with stderr content.

BUG 3 — logs --last crashes if value is not a valid integer
File: src/cli.py — _cmd_logs()
Severity: MEDIUM — unhandled ValueError crashes the command
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  User types:  logs --last abc
  Code runs:   n = int(intent.flags.get("last", 50))

WHAT'S NEEDED:

BUG 4 — disk_usage("/") crashes on Windows
File: modules/system.py — info()
Severity: HIGH on Windows — unhandled exception
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  The code always does:  disk = psutil.disk_usage("/")
  On Linux and macOS "/" is the root. Works fine.
  On Windows, "/" is not a valid path. psutil raises FileNotFoundError.
  The info() function crashes with an unhandled exception,
  and the terminal shows "Internal error: [WinError 3] The system
  cannot find the path specified: '/'"

  This is one of the most-used commands. Typing "system info"
  on Windows would always fail.

WHAT'S NEEDED:
  import platform at top (already imported), then:
  disk_root = "C:\\" if os_type == "windows" else "/"
  disk = psutil.disk_usage(disk_root)

BUG 5 — _build_skill_triggers() reads every YAML file from disk
         on every single command the user types
File: src/cli.py
Severity: MEDIUM — performance issue + unnecessary I/O
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  Every time the user presses Enter, _execute_command() is called.
  The first thing it does is call _build_skill_triggers() which:
    1. Opens the skills/ directory
    2. Iterates every .yaml file
    3. Opens and parses each one with yaml.safe_load()
    4. Builds a dictionary of {phrase: skill_name}
    5. Returns it and throws it away

  If you have 4 skill files and type 100 commands in a session,
  that's 400 file reads and 400 YAML parses for no reason.
  Skills don't change while the program is running.

WHAT'S NEEDED:
  Cache the result in self._skill_triggers on first call.
  Return the cached version on all subsequent calls.
  Only rebuild if the user explicitly runs a "skills reload" command.

BUG 6 — count_online() returns 0 for ADB-mode devices
File: src/discovery.py
Severity: MEDIUM — banner always shows wrong device count
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  When a device has no IP (ADB mode), discovery sets its status to:
  "no-ip (adb mode)"

  But count_online() only counts devices with status == "online":
    return sum(1 for d in self._cache.values() if d.get("status") == "online")

  ADB devices never match "online" so they're never counted.
  The startup banner says "0 device(s) online" even when your
  phone is plugged in via USB and fully working.

WHAT'S NEEDED:
  Count both "online" AND "no-ip (adb mode)" as online.
  "adb" status in cli.py's device table is also treated as online
  visually — the count should reflect the same logic.

PART 2 — SECURITY ISSUES
================================================================================

────────────────────────────────────────────────────────────────────────────────
SECURITY 1 — Agent binds to 0.0.0.0 by default (exposed to entire network)
File: agent/agent.toml
Severity: CRITICAL
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  listen_host = "0.0.0.0" means the agent listens on EVERY network
  interface — your home Wi-Fi, ethernet, VPN, everything.

  Anyone on the same Wi-Fi network as the target device can reach
  port 7070 and attempt commands. In a cafe, university, or shared
  office — that's hundreds of people.

  The config says "Change to Tailscale IP for security" as a comment.
  But the default should be the SAFE option. Security defaults must be
  restrictive, not permissive. Nobody reads comments until after they
  get compromised.

REAL ATTACK SCENARIO:
  1. User installs JARVIS on their laptop at a university
  2. Laptop is on university Wi-Fi
  3. Agent binds to 0.0.0.0:7070
  4. Any student with a port scanner finds it
  5. Sends POST /command with a guessed or brute-forced token
  6. Gets access to screenshot, file listing, and potentially more

WHAT'S NEEDED:
  Default listen_host = "127.0.0.1"
  Document: "Change to your Tailscale IP (100.x.x.x) to allow remote TCC connections"
  Add startup error if token is still "CHANGE_ME_strong_secret_token_here"


────────────────────────────────────────────────────────────────────────────────
SECURITY 2 — Empty auth token silently enables unauthenticated access
File: agent/agent.py — require_auth decorator
Severity: CRITICAL
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  The auth decorator does this:
    if not AUTH_TOKEN:
        return f(*args, **kwargs)  # No token configured — dev mode

  If auth_token is empty or missing in agent.toml, the agent accepts
  ALL requests from ANYONE with NO authentication check.

  This is labeled "dev mode" but there is no visible warning when the
  agent starts in this state. The startup message just says:
    Auth : NONE (dev mode)

  But most users won't notice a line in a wall of startup text.

REAL SCENARIO:
  User forgets to set auth_token, runs agent.
  Anyone who can reach port 7070 has full agent access.
  No token required. No error. No alert.

WHAT'S NEEDED:
  On startup, if AUTH_TOKEN is empty:
  - Print a loud multi-line WARNING
  - Refuse to bind to anything except 127.0.0.1
  - Optionally exit with an error unless --allow-no-auth flag is passed


────────────────────────────────────────────────────────────────────────────────
SECURITY 3 — Agent uses HTTP, not HTTPS — tokens travel in plaintext
File: src/router.py, agent/agent.py
Severity: HIGH
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  All communication between TCC client and agents is:
    http://{ip}:{port}/command

  The auth token is sent in every request:
    "Authorization": f"Bearer {auth_token}"

  Over plain HTTP, this token is transmitted in cleartext.
  On a local network, any device doing passive packet capture
  (even without ARP spoofing) can read it.

  Once the token is captured, the attacker has permanent agent access
  because tokens don't rotate and there's no session concept.

WHY TAILSCALE DOESN'T FULLY SOLVE THIS:
  Tailscale encrypts traffic between peers. But if the agent is bound
  to 0.0.0.0 (Bug 1), traffic can also arrive over the LAN interface
  where it is NOT encrypted.

WHAT'S NEEDED:
  Option 1 (simple): Bind only to Tailscale IP. Tailscale encrypts.
  Option 2 (proper): Use HTTPS with self-signed cert generated on first run.
  Option 3 (good enough): At minimum, hash the token with HMAC per-request
                           so replaying a captured auth header doesn't work.


────────────────────────────────────────────────────────────────────────────────
SECURITY 4 — No rate limiting on agent endpoints
File: agent/agent.py
Severity: HIGH — enables token brute force
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  The /command endpoint has no rate limiting.
  An attacker who can reach the agent can try unlimited tokens per second.

  A 32-character hex token (recommended) is computationally unbrute-forceable.
  But the default token in the config is:
    "CHANGE_ME_strong_secret_token_here"

  If any user leaves this as-is (which happens), an attacker only needs
  to try a handful of obvious defaults. No lockout. No delay. No alarm.

WHAT'S NEEDED:
  At startup: reject if token matches the default placeholder exactly.
  Simple rate limit: track failed auth attempts per IP.
  After 5 failures in 60 seconds, block that IP for 5 minutes.
  Log all auth failures with timestamps and source IPs.


────────────────────────────────────────────────────────────────────────────────
SECURITY 5 — phone.py run() passes user commands through shlex
             but still gives full phone shell access
File: modules/phone.py — run()
Severity: HIGH — intended but dangerous feature with no guardrails
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  The run() action lets you execute arbitrary shell commands on the phone:
    phone run "rm -rf /sdcard/"
    phone run "cat /sdcard/WhatsApp/Databases/msgstore.db"

  shlex.split() does prevent shell injection (no &&, ;, | etc.),
  but it doesn't prevent dangerous commands themselves.

  In the agent permission system, "run" is in denied_actions by default.
  But in the ADB mode (router.py → phone.py), there is NO permission check.
  The PhoneModule.run() method is called directly with no ACL.

REAL SCENARIO:
  User types: phone run "rm -rf /sdcard/DCIM"
  No confirmation. No warning. Immediately executes.
  All photos deleted.

WHAT'S NEEDED:
  Add a confirmation prompt for "run" on any target:
  "Execute 'rm -rf /sdcard/DCIM' on phone? [y/N]"
  And/or add a local ACL for ADB mode mirroring the agent permission system.


────────────────────────────────────────────────────────────────────────────────
SECURITY 6 — Log injection via crafted command strings
File: src/logger.py — log_command()
Severity: LOW-MEDIUM — can pollute or forge log entries
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  Log entries are built by string concatenation:
    line = f"[{ts}] {level:<5}  {command:<35} {status:<10} {latency:>5}ms"

  If a user types a command containing newlines (\n) or log-format
  strings like "[2026-03-09 12:00:00] ERROR  laptop run", those
  characters go directly into the log file uncleaned.

  Result: a malicious command string can inject fake log entries.

  Example:
    Type: phone battery\n[2026-03-09 12:00:00] ERROR  system shutdown success
    The log file now contains a fake ERROR entry for a shutdown that never happened.

WHAT'S NEEDED:
  Sanitize the command string before logging:
    command_clean = repr(command) or command.replace("\n", "\\n").replace("\r", "\\r")


────────────────────────────────────────────────────────────────────────────────
SECURITY 7 — Skill path traversal not sanitized (in original code)
File: src/cli.py — _run_skill()
Severity: MEDIUM — can read arbitrary YAML files if misconfigured
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  The original _run_skill() builds the path like this:
    skill_file = os.path.join(skills_dir, f"{skill_name}.yaml")

  skill_name comes from a YAML file's "trigger" list.
  If a maliciously crafted skill YAML has a step that triggers
  another skill named "../../config", the path becomes:
    skills_dir/../../config.yaml

  This could load config.toml-adjacent files as if they were skills,
  exposing the auth token via the steps list rendering.

WHAT'S NEEDED:
  os.path.basename(skill_name) to strip any path components.
  Validate the resulting path stays inside the skills/ directory.
  This fix was already applied in the corrected version.


────────────────────────────────────────────────────────────────────────────────
SECURITY 8 — Flask runs in development mode (not production-safe)
File: agent/agent.py — start_agent()
Severity: MEDIUM — development server not designed for real traffic
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  app.run(host=LISTEN_HOST, port=LISTEN_PORT, debug=False, use_reloader=False)

  debug=False is correct — good.
  But Flask's built-in server (Werkzeug) is explicitly documented as
  NOT suitable for production. It is single-threaded by default, which means:

  1. Only ONE request can be processed at a time.
  2. A slow command (e.g. screenshot taking 2 seconds) blocks ALL others.
  3. If TCC sends a broadcast to 3 devices, and one agent is slow,
     the others wait in queue.

WHAT'S NEEDED:
  Use a production WSGI server. Options that are free and lightweight:
    - waitress (pip install waitress) — pure Python, works on Windows too
    - gunicorn (Linux/macOS only)

  Replace app.run() with:
    from waitress import serve
    serve(app, host=LISTEN_HOST, port=LISTEN_PORT, threads=4)


────────────────────────────────────────────────────────────────────────────────
SECURITY 9 — Auth tokens stored in plaintext config files with no permissions check
File: config.toml, agent/agent.toml
Severity: MEDIUM
────────────────────────────────────────────────────────────────────────────────

WHAT HAPPENS:
  Both config files store the auth token in plain readable text.
  On Linux/macOS these files default to 644 (world-readable) unless
  the user manually sets 600. On Windows, permissions are rarely set.

  If the machine is compromised, shared, or the repo is accidentally
  pushed to GitHub (happens constantly), the token is exposed.

WHAT'S NEEDED:
  On startup, check config.toml and agent.toml file permissions.
  On Linux/macOS: warn if permissions are more permissive than 600.
  Alternatively: support reading token from environment variable:
    auth_token = env:TCC_AUTH_TOKEN
  So the token never lives in a file at all.

MISSING FEATURE 3 — Device Pairing / Approval Flow
Reference: moltworker — Device Pairing section
────────────────────────────────────────────────────────────────────────────────

WHAT OPENCLAW HAS:
  When a new device tries to connect, it enters a pending state.
  An admin must approve it via the admin UI before it can send commands.
  This prevents unauthorized devices from connecting even with a token.

CURRENT STATE IN GINTAMA-BOT:
  Token-only. If the token matches, you're in. There's no concept of
  "this is a new device I haven't seen before, let me verify it."
  Anyone who obtains the token from any source gets full access immediately.

HOW TO BUILD IT FOR FREE:
  Each device generates a unique device_id on first run (UUID).
  Store known/approved device_ids in a local JSON file on the server.

  Flow:
    1. New device connects with token + device_id
    2. If device_id not in approved list → status "pending", command rejected
    3. Run "devices --pending" in TCC to see waiting devices
    4. Run "devices --approve <device_id>" to approve
    5. Device_id added to approved list → future commands accepted

  No cloud. No admin panel needed. Just terminal commands.
  Persistent across restarts using a simple JSON file.

  Benefit: agent can push events to TCC without being asked.

  The container can sleep after a configurable period of inactivity
  to save compute costs. Wake happens on next request.

CURRENT STATE IN GINTAMA-BOT:
  Agent runs forever. No idle detection. No auto-sleep.
  Not a huge issue for personal devices, but on a server or secondary
  laptop you might want the agent to use minimal resources when idle.

HOW TO BUILD IT FOR FREE:
  Add idle_timeout_minutes to agent.toml.
  Track last_command_time in the agent.
  After timeout: reduce check interval, log "agent idle", optionally
  send a notification to TCC that it's going into low-power mode.
  On next command received: resume normally.

PART 4 — CODE QUALITY ISSUES (not bugs, but need improvement)
================================================================================

────────────────────────────────────────────────────────────────────────────────
QUALITY 1 — No input length validation anywhere
────────────────────────────────────────────────────────────────────────────────

  User can type an arbitrarily long command string.
  App name, notify message, file paths — all unbounded.
  A 10MB "message" in the notify field would be accepted and logged.

  Add max length checks:
  - Command string: 512 chars
  - Notify message: 500 chars (also limits SMS/notification payloads)
  - File paths: 260 chars (Windows MAX_PATH limit)
  - App name: 100 chars


────────────────────────────────────────────────────────────────────────────────
QUALITY 2 — No startup validation of config.toml
────────────────────────────────────────────────────────────────────────────────

  TCC starts even if config.toml has:
  - Invalid device type values
  - Ports outside 1-65535
  - Token still set to the placeholder value
  - IP addresses that are syntactically invalid

  The errors only appear later when you try to use a broken device.

  Add a config validator that runs on startup:
  - Required fields present
  - Port is a valid integer in range
  - IP is a valid IPv4 if transport != "adb"
  - Token is not the placeholder default
  - Device type is one of: android | windows | linux | darwin


────────────────────────────────────────────────────────────────────────────────
QUALITY 3 — Scheduler resolution is 30 seconds — jobs can drift by up to 30s
File: src/scheduler.py — _loop()
────────────────────────────────────────────────────────────────────────────────

  The loop sleeps 30 seconds between checks.
  A task scheduled for 07:00 might run at 07:00:29.
  Over time this is fine, but it's sloppy.

  Better: sleep until the next whole minute boundary:
    now = datetime.now()
    sleep_seconds = 60 - now.second
    time.sleep(sleep_seconds)

  Now the loop fires exactly at each minute mark. Drift is < 1 second.


────────────────────────────────────────────────────────────────────────────────
QUALITY 4 — No graceful shutdown — agent leaves port open on crash
File: agent/agent.py
────────────────────────────────────────────────────────────────────────────────

  If the agent crashes (unhandled exception), or is killed with Ctrl+C,
  the port may remain in TIME_WAIT state for 60-120 seconds.
  Next restart attempt fails with "Address already in use".

  Add:
    app.run(..., use_reloader=False)  ← already done, good
    Also add socket SO_REUSEADDR option before binding.
    And catch KeyboardInterrupt in start_agent() to log clean shutdown.


────────────────────────────────────────────────────────────────────────────────
QUALITY 5 — _build_skill_triggers silently skips malformed YAML
File: src/cli.py
────────────────────────────────────────────────────────────────────────────────

  The except block in _build_skill_triggers is bare:
    except Exception:
        pass

  If a YAML file has a syntax error, the skill is silently skipped.
  The user types "good morning", nothing happens, no error message.
  They have no idea why the skill isn't triggering.

  At minimum: log the exception at WARNING level.
  Better: print a one-time warning on startup for each invalid skill file.


────────────────────────────────────────────────────────────────────────────────
QUALITY 6 — No version mismatch detection between TCC client and agents
────────────────────────────────────────────────────────────────────────────────

  Agent /health returns "version": "2.0".
  TCC client never reads this version field.

  If you update TCC client to v3.0 with new command schema but agents
  are still running v2.0, commands may fail with cryptic errors.

  Add version check: if agent major version != client major version,
  print a warning: "Agent at 100.98.44.7 is v2.0, TCC is v3.0.
  Update the agent: python main.py --agent"
