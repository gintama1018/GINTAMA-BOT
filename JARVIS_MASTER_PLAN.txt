━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  You said: "open camera and take my selfie" — and it DID it.

  This is NOT a command parser. This is an AGENT LOOP. The difference:

  COMMAND PARSER (what TCC currently has):
    "open camera" → maps to ONE command → executes it → done
    "open camera and take my selfie" → FAILS (two steps, no plan)

  AGENT LOOP (what OpenClaw has, what JARVIS needs):
    "open camera and take my selfie"
      → AI THINKS: what steps are needed?
          Step 1: phone launch camera
          Step 2: wait 2 seconds (camera needs to open)
          Step 3: phone screenshot (selfie = front camera screenshot)
          Step 4: pull screenshot to local machine
          Step 5: send image back to you on Telegram
      → AI EXECUTES each step as a "tool call"
      → AI CHECKS: did each step succeed?
      → AI REPORTS: sends you the image on Telegram

  How it works technically:
    Gemini 2.5 Flash has NATIVE function/tool calling.
    You define your TCC commands as "tools" in the API call.
    Gemini decides which tools to call and in what order.
    The result of each tool call goes back to Gemini.
    Gemini decides what to do next.
    Loop continues until task is complete.

  This is the CORE of everything. Build this first.

┌─────────────────────────────────────────────────────────────────────────────┐
│ SECTION 1 — THE CORE: AGENT LOOP                                            │
└─────────────────────────────────────────────────────────────────────────────┘

  OpenClaw:  Pi-agent-core runtime, tool streaming, lifecycle events
             Uses Claude Opus 4.6 (requires $100-200/mo Pro/Max subscription)

  JARVIS:    Python agent loop engine
             Uses Gemini 2.5 Flash (free tier) → Ollama mistral (offline)
             Tools defined as Gemini function_declarations
             Loop: message → LLM → tool call → execute → LLM → tool call → ...
             Until LLM returns text with no tool call = task complete

  File:      src/agent_loop.py (NEW — core of everything)

  Key design:
    - Serialized per session (one loop at a time per user/session)
    - Each tool call is a real function execution on the device
    - Tool results go back to LLM as function_response
    - Max 20 tool call iterations per turn (safety limit)
    - Timeout: 120 seconds per agent turn
    - On timeout: send "Still working on it..." and continue
    - On error: catch, log, report clean message to user

  Tool definitions (Gemini function_declarations format):
    phone_launch     - launch app on phone
    phone_screenshot - take screenshot of phone screen
    phone_battery    - get battery level
    phone_volume     - set volume
    phone_lock       - lock phone
    phone_pull       - transfer file from phone to PC
    phone_push       - transfer file from PC to phone
    phone_notify     - send notification to phone
    system_info      - get local system info
    system_screenshot - screenshot local screen
    system_open      - open app/file locally
    system_run       - run shell command (requires confirmation)
    system_notify    - local desktop notification
    device_info      - get info from any registered device
    device_run       - run command on registered device
    file_ls          - list files
    file_read        - read file contents
    file_write       - write file (requires confirmation)
    browser_open     - open URL in headless browser
    browser_screenshot - screenshot a URL
    web_search       - search the web (SerpAPI free tier or DuckDuckGo)
    schedule_task    - schedule a recurring task
    send_message     - send message back to user on their channel

┌─────────────────────────────────────────────────────────────────────────────┐
│ SECTION 5 — SECURITY MODEL (MIRRORS OPENCLAW EXACTLY)                       │
└─────────────────────────────────────────────────────────────────────────────┘

  ── 5A. GATEWAY BINDING ───────────────────────────────────────────────────────

    JARVIS Gateway MUST:
      Default: bind to 127.0.0.1 ONLY
      To allow Tailscale access: bind to Tailscale IP (100.x.x.x)
      NEVER bind to 0.0.0.0 by default
      Config: gateway.bind = "loopback" | "tailscale" | "custom"
      Warning printed on startup if bind is anything other than loopback

  ── 5B. DM PAIRING ────────────────────────────────────────────────────────────

    Exactly mirrors OpenClaw's pairing model:

    Unknown sender → receives pairing code (6-digit, expires 1 hour)
    You approve:  jarvis pairing approve telegram abc123
    Approved sender ID written to: ~/.jarvis/credentials/{channel}-allowlist.json

    Config options:
      dmPolicy = "pairing"   (default — unknown senders get code)
      dmPolicy = "allowlist" (unknown senders silently blocked)
      dmPolicy = "open"      (anyone — NEVER USE unless testing)
      dmPolicy = "disabled"  (ignore all DMs)

    Max 3 pending pairing requests per channel.
    jarvis pairing list telegram — see pending requests
    jarvis pairing revoke telegram <id> — remove approved sender

  ── 5C. SESSION ISOLATION ────────────────────────────────────────────────────

    Each sender gets an ISOLATED session:
      - Separate conversation history
      - Separate memory
      - Cannot see other users' sessions
      - Cannot trigger actions from another user's context

    session.dmScope = "per-channel-peer"
    This is the SECURE DEFAULT. Never use "main" if multiple people can DM.

  ── 5D. GROUP CHAT SECURITY ──────────────────────────────────────────────────

    requireMention = True (default)
      JARVIS only responds in group chats when @mentioned
      Prevents ambient listening in group conversations

    groupAllowFrom: list of allowed user IDs in groups
      Even in allowed groups, filter who can trigger the bot

    Group allowlist: only respond in specific groups/servers/channels

  ── 5E. TOOL POLICY ──────────────────────────────────────────────────────────

    Per-agent tool profiles:

    TRUSTED (your own messages):
      All tools enabled
      run/exec requires confirmation for destructive commands
      Full device access

    FAMILY/SHARED (approved friends):
      browser: disabled
      run/exec: disabled
      file write: disabled
      read-only access to device info, screenshots

    PUBLIC (open DMs — not recommended):
      ONLY messaging tools
      No device access
      No file access
      No browser

    Sandbox mode (optional, Docker required):
      Wrap run/exec in Docker container with --network=none
      Prevents any network access from executed code
      Prevents filesystem access beyond container

  ── 5F. PROMPT INJECTION PROTECTION ─────────────────────────────────────────

    Mirrors OpenClaw's guidance exactly:

    Defense in depth:
      1. Only trusted senders can trigger the bot (pairing/allowlists)
      2. Tool allowlist limits blast radius
      3. Confirmation required for destructive actions
      4. All tool calls logged with sender identity
      5. Web content treated as untrusted (separate reader context)

    System prompt injection:
      "Ignore all previous instructions" → Gemini 2.5 Flash resistant
      Logged and flagged if detected
      When Ollama is used: enable model-level injection resistance prompts

    Content injection (web pages, emails, attachments):
      Reader agent pattern: separate "reader" agent with NO tools
      Reader summarizes content → summary passed to main agent
      Main agent never directly reads untrusted external content

  ── 5G. CREDENTIAL SECURITY ──────────────────────────────────────────────────

    All credentials stored in: ~/.jarvis/
    File permissions: 700 for directory, 600 for all files
    Set on install: chmod 700 ~/.jarvis && chmod 600 ~/.jarvis/*.json

    Never stored in code or Git:
      GEMINI_API_KEY — env var
      TELEGRAM_BOT_TOKEN — env var
      DISCORD_BOT_TOKEN — env var
      JARVIS_AUTH_TOKEN — env var
      All channel tokens — env vars

    startup check: if token == placeholder → refuse to start, print error

    Token rotation checklist (printed by: jarvis security rotate):
      1. Generate new token
      2. Update env var
      3. Restart JARVIS
      4. Update remote agent configs
      5. Verify old token no longer works

  ── 5H. JARVIS SECURITY AUDIT COMMAND ────────────────────────────────────────

    OpenClaw has: openclaw security audit
    JARVIS has:   jarvis security audit

    Checks:
      ✓ Gateway bind mode (warns if not loopback)
      ✓ Auth token not placeholder, length >= 32 chars
      ✓ Config file permissions (should be 600)
      ✓ ~/.jarvis directory permissions (should be 700)
      ✓ DM policies for all channels (warns if any = "open")
      ✓ Group policies (warns if requireMention = False)
      ✓ Tool policies (warns if run/exec enabled with open DMs)
      ✓ Rate limiting enabled
      ✓ Pairing allowlists reviewed recently
      ✓ Agent token not default placeholder
      ✓ No tokens found in logs or session transcripts
      ✓ Tailscale running (if tailscale transport configured)

    File: src/security_audit.py (NEW)

  ── 5I. RATE LIMITING ────────────────────────────────────────────────────────

    Per-sender rate limits:
      10 messages per minute (configurable)
      100 messages per day (configurable)
      Failed auth: 5 attempts in 60s → block IP for 5 minutes

    Stored in: in-memory dict + optional Redis if you have it
    On limit hit: send message "Too many requests. Try again in X minutes."

  ── 5J. AUDIT LOGGING ────────────────────────────────────────────────────────

    Every action logged with:
      timestamp, sender_id, channel, message, tool_calls[], results, latency

    Redaction: API keys, tokens, passwords auto-redacted from logs
    Retention: 7 days by default (configurable)
    Location: ~/.jarvis/logs/audit-YYYY-MM-DD.jsonl

    jarvis logs --last 20 — view recent
    jarvis logs --sender telegram:123456 — filter by sender
    jarvis logs --since 1h — filter by time

    Session isolation: each sender_id + channel = isolated session

  ── 6B. MEMORY / COMPACTION ──────────────────────────────────────────────────

               Original messages preserved in DB
